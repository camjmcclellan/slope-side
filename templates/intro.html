How It Works

The autonomous daily loop

Every morning the system wakes up and runs without any input. Here is exactly what happens.

Morning Review at 8am

A scheduled job fires at 8am. The system pulls the last 24 hours of email history, reads every reply that came in, and builds a performance report for the two prompt variants that were tested the previous day.

  • Reads reply rates per variant
  • Passes full reply text to the LLM
  • LLM generates two improved system prompt variants
  • New variants saved for today's outreach
[scheduler] 08:00:00 Morning Review # Pull last 24hrs of performance data Variant A: 12 sent, 2 replies Variant B: 11 sent, 0 replies # LLM analyzes and generates new prompts variant_a: "You are a direct..." variant_b: "You are a consultative..." Saved variant A (id=7), variant B (id=8)

Personalized emails at scale

The system selects 50% of leads at random and splits them evenly between the two new prompt variants. It calls the LLM once per lead, generating a unique email using each lead's name, company, title, industry, and keywords.

  • One LLM call per lead
  • Uses real lead data for personalization
  • A/B split for continued optimization
  • All emails written and stored before any are sent
[1/23] Generating for Sarah Kim (Acme Corp) subject: "Quick thought on your ops team" scheduled: 08:01 UTC [2/23] Generating for James Obi (RetailCo) subject: "Scaling retail ops affordably" scheduled: 08:22 UTC # 23 emails queued across 8hr window

Staggered sending over 8 hours

Emails are not blasted all at once. They are staggered evenly across an 8-hour window starting 1 minute after the review completes. A background job checks every minute and fires emails when their scheduled time arrives.

  • First email sends at 8:01am
  • Rest spaced evenly through 4pm
  • Mimics human sending behavior
  • Avoids spam filters triggered by bulk sends
[scheduler] dispatch_scheduled_emails() # Runs every 60 seconds 08:01 Sent plan id=41 → Sarah Kim 08:22 Sent plan id=42 → James Obi 08:43 Sent plan id=43 → Priya Shah 09:04 Sent plan id=44 → Marcus Lee ...

Reply tracking and matching

Every 15 minutes the system checks the inbox. It matches replies to the original outbound email using the Message-ID and In-Reply-To email headers, strips quoted reply text, and stores the clean response in the history record. This data feeds directly into the next morning's review.

  • Checks all mail from the last 24 hours
  • Primary match via In-Reply-To header
  • Fallback match via subject line
  • Deduplicates to avoid double-counting
[scheduler] check_inbox() 15min interval Reply from james@retailco.com In-Reply-To: <msg-id-42@scaleme> Matched → history id=42 # Stripped quoted text, stored reply body responses: [{ "from": "james@retailco.com", "body": "Hey, curious to learn more..." }]
Technical Stack

What is under the hood

Built in Python with a focus on simplicity, reliability, and genuine autonomous behavior rather than simulated automation.

Python / Flask Google Gemini via OpenRouter SQLite + WAL mode APScheduler Gmail SMTP IMAP reply parsing Message-ID header tracking Server-Sent Events (SSE) A/B prompt optimization loop Caddy reverse proxy AWS EC2


What makes this genuinely autonomous
  • No cron job triggers a human-written script. The AI decides what to write based on what worked yesterday.
  • The prompt that generates emails is itself generated by AI, rewritten daily based on live performance data.
  • The system reads its own email inbox, interprets replies in context, and feeds that signal back into its decision loop.
  • A human can go weeks without touching this and it continues to operate, learn, and improve.
